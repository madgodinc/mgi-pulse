# Benchmarks

Reference numbers from one dev box. The intent is to spot regressions
between releases, not to compete with specialised processors.

## Reference machine

- CPU: Intel i5-12400F (6c/12t, ~4.4 GHz turbo)
- RAM: 48 GB DDR4-3200
- Disk: NVMe SSD, ext4
- OS: Pop!_OS 24.04, Linux 6.11

## End-to-end indexing

Target fixture: 2 GB / 11 M-record synthetic NDJSON (generated by
`bench/gen-ndjson.sh`).

| Step | Wall time | Throughput |
|---|---|---|
| Indexer pass (line/time/severity arrays) | ~2.8 s | ~700 MB/s, ~4.0 M records/s |
| Plus schema warmup (first 10k records) | ~2.85 s | — |
| First UI paint after indexer | < 100 ms (informal) | — |

Reproduce on your hardware:

```sh
./bench/gen-ndjson.sh /tmp/bench.ndjson 11000000     # ~1 minute, ~2 GB output
cargo run --release -p parse-bench --bin indexer-bench -- /tmp/bench.ndjson
```

The bursty variant `gen-ndjson-bursty.sh` produces the same shape with
time-varying severity distribution — that's what the README hero
screenshot uses.

## Parser hot-path

From `parse-bench` (separate binary, measures `ts` + `level` extraction
only, same 2 GB file):

| Strategy | Throughput | Notes |
|---|---|---|
| `memchr` line splitter | ~12 GB/s | floor; everything else has to beat this |
| serde-borrow with `&RawValue` ts + `&str` level | ~905 MB/s | shipped default |
| `simd-json` mutable parse | ~1.1 GB/s | wins ~20%; not adopted, would add build dep |

Cost of the per-format dispatch match in `LogFormat::parse_ts_level`
doesn't show up in profiles — < 1% of total indexer time on the
single-format case.

## Memory

Index footprint is roughly **15-20 MB per million records** (three
parallel arrays: 16 bytes per `LineLoc` + 8 bytes per `ts_micros` +
1 byte per severity). The 11 M-record fixture lands at ~180 MB
resident.

`Box<[u8]>` from the stream path adds the bytes themselves; mmap
files cost nothing beyond the index because record payloads stay in
the kernel page cache.

## What's NOT measured here

- **First-paint latency under sustained follow load.** The follow
  worker batches into the channel; the UI drains up to 4096 per
  tick. We don't have a measurement of perceived latency at
  10k-events/second sustained writes.
- **Memory at 10 GB+.** The index scales linearly with record count.
  100 M records is roughly 1.5-2 GB resident; the engine doesn't
  evict.
- **Cold cache.** All numbers above are second-run with the page
  cache warm. First run on a freshly-mounted disk is dominated by
  disk IO, not parse.
- **Non-Linux platforms.** macOS / Windows builds compile in CI but
  haven't been profiled.
